## §1 教案資訊

| 欄位 | 內容 |
|------|------|
| **對象** | 有程式基礎但沒 Go/MCP 經驗的開發者；想快速掌握 MCP Server 開發的工程師 |

## §2 學習成果

結訓後學員應能：

1. **能說明** MCP 協定的核心概念與三種 capabilities（Resources、Tools、Prompts）的差異
2. **能建立** 最小可執行的 MCP Server 並與 Claude Desktop 連接
3. **能實作** Tool capability，定義輸入參數與回傳結果
4. **能實作** Resource capability，提供靜態或動態資料給 Client
5. **能實作** Prompt capability，建立可重用的提示詞模板
6. **能切換** Stdio 與 HTTP 兩種傳輸模式
7. **能實作** HTTP Server 認證機制（Basic Auth、Bearer Token）
8. **能整合** 三種 capabilities 完成一個 MVP 等級的天氣查詢服務
9. **能除錯** 常見的 MCP Server 連線與執行問題

### 依賴標記

- `⟵ 需完成 UXX`：表示此單元依賴前置單元，需先完成前置單元

### 單元編號格式

- `U00`：課前準備
- `U01-U09`：核心單元
- 每單元包含：為什麼先教這個、對應素材、核心觀念、動手做、踩坑提示、驗收標準

### 教案與範例的關係

| 文件 | 閱讀目的 |
|------|---------|
| 各單元 MD（`units/UXX-*.md`） | 理解原理、架構、流程與關鍵邏輯 |
| 範例程式碼（`examples/UXX-*/`） | 取得完整可執行的實作（Single Source of Truth） |

- 教案 MD 中的程式碼**刻意省略非關鍵部分**（以 `// ...` + 被省略內容首行前 10-30 字 示意），聚焦架構與重點

- 建議閱讀順序：先讀教案理解「為什麼」→ 再看 `examples/` 理解「怎麼做」

### 驗證指令

```bash
# 1. 確認 Go 版本
go version
# 預期輸出：go version go1.21.x (或更高)

# 2. 確認 Go 環境變數
go env GOPATH
# 預期輸出：顯示路徑（如 /Users/xxx/go）

# 3. 建立測試專案
mkdir -p ~/test-go && cd ~/test-go
go mod init test
echo 'package main; import "fmt"; func main() { fmt.Println("Go is ready!") }' > main.go
go run main.go
# 預期輸出：Go is ready!

# 4. 清理測試專案
cd ~ && rm -rf ~/test-go
```

### 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| `go: command not found` | Go 未安裝或 PATH 未設定 | 重新安裝 Go，確認 `export PATH=$PATH:/usr/local/go/bin` 已加入 shell 設定 |
| `go version` 顯示舊版本 | 系統有多個 Go 版本 | 執行 `which go` 確認路徑，移除舊版或調整 PATH 順序 |

### U01｜Hello MCP Server（90 min）

#### ① 為什麼先教這個？

MCP Server 是與 AI 助手溝通的橋樑。先讓一個最小的 Server 跑起來，確認環境正確，才有信心繼續學習更複雜的功能。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 範例程式碼 | `examples/U01-hello/main.go` | 最小 MCP Server |
| 設定範例 | `examples/U01-hello/claude_desktop_config.json` | Claude Desktop 設定 |
| 官方文件 | https://modelcontextprotocol.io/docs/develop/build-server | 建置指南 |

#### ③ 核心觀念

1. **MCP 是什麼**：Model Context Protocol，讓 AI 助手能呼叫外部工具、讀取資源的標準協定
2. **Server vs Client**：Server 提供功能，Client（如 Claude）呼叫功能
3. **Capabilities 三類**：Tools（函式呼叫）、Resources（檔案類資料）、Prompts（提示詞模板）
4. **Transport 方式**：Stdio（標準輸入輸出）、HTTP
5. **JSON-RPC 2.0**：MCP 使用的通訊協定格式
6. **Implementation 結構**：定義 Server 名稱與版本
7. **Logging 規則**：Stdio 模式下禁止使用 `fmt.Println()`，只能用 `log.Println()`（輸出到 stderr）

#### ④ 動手做

**[必做]**
1. 建立專案目錄並初始化 Go module
2. 撰寫最小 MCP Server（僅回應 initialize 請求）
3. 用 `go run` 啟動 Server，觀察輸出
4. 設定 Claude Desktop 連接此 Server

**[延伸]**
1. 在 Server 加入 `log.Println()` 紀錄每次請求
2. 嘗試修改 Server 名稱，觀察 Claude Desktop 顯示變化

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Claude Desktop 無法找到 Server | config.json 路徑錯誤或 JSON 格式錯誤 | 確認路徑為絕對路徑，用 JSON 驗證器檢查格式 |
| Server 啟動後立即結束 | Stdio 模式需持續監聽輸入 | 確認使用 `server.Run()` 而非直接結束 |
| `fmt.Println` 導致 Client 連線失敗 | Stdio 模式下 stdout 用於 JSON-RPC 通訊 | 改用 `log.Println()`（輸出到 stderr） |

#### ⑥ 驗收標準（DoD）

- [ ] 執行 `go run main.go` 不報錯
- [ ] Claude Desktop 能顯示此 Server 於連接器清單
- [ ] Server log 顯示收到 initialize 請求

### U02｜MCP Inspector（45 min）

#### ① 為什麼先教這個？

MCP Inspector 是官方提供的測試工具，能直接與 Server 互動、查看 JSON-RPC 訊息。學會使用後，除錯效率大幅提升。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 官方文件 | https://modelcontextprotocol.io/docs/tools/inspector | Inspector 使用指南 |
| 安裝指南 | `docs/setup/mcp-inspector.md` | 本地安裝步驟 |

#### ③ 核心觀念

1. **Inspector 是什麼**：官方 MCP 測試與除錯工具
2. **啟動方式**：`npx @modelcontextprotocol/inspector`
3. **連線模式**：支援 Stdio 和 HTTP 兩種
4. **功能面板**：Tools、Resources、Prompts 分頁測試
5. **JSON-RPC 檢視**：查看完整的請求/回應訊息
6. **Logs 面板**：查看 Server 端的 log 輸出

#### ④ 動手做

**[必做]**
1. 安裝 Node.js 18+
2. 使用 `npx @modelcontextprotocol/inspector` 啟動
3. 連接 U01 的 Hello Server
4. 測試 initialize 請求

**[延伸]**
1. 觀察 JSON-RPC 的完整格式
2. 嘗試手動輸入錯誤的 method 名稱，觀察錯誤回應

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| `npx` 指令找不到 | Node.js 未安裝 | 安裝 Node.js 18+ |
| 連線失敗 | Server 路徑錯誤 | 確認 `go run` 指令的完整路徑 |
| 無法看到 Tools | Server 未註冊 Tool | 先完成 U03 再測試 |

#### ⑥ 驗收標準（DoD）

- [ ] Inspector 能成功啟動
- [ ] 能連接本地 MCP Server
- [ ] 能查看 JSON-RPC 訊息內容

### U03｜Tool Capability（120 min）

#### ① 為什麼先教這個？

Tool 是 MCP 最常用的功能——讓 AI 能「做事」。學會 Tool 後，就能讓 Claude 執行任意自定義函式。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 範例程式碼 | `examples/U03-tools/main.go` | 含 add、greet 兩個 Tool |
| 輸入結構 | `examples/U03-tools/schema.go` | 輸入參數定義 |

#### ③ 核心觀念

1. **Tool 定義**：名稱、描述、輸入 Schema
2. **mcp.AddTool()**：註冊 Tool 到 Server
3. **Input Schema**：使用 struct tag 定義 JSON Schema
4. **Tool Handler**：處理請求並回傳結果
5. **CallToolResult**：回傳內容（TextContent、ImageContent 等）
6. **錯誤處理**：回傳錯誤訊息而非 panic

#### ④ 動手做

**[必做]**
1. 新增一個 `add` Tool，接收兩個數字並回傳相加結果
2. 新增一個 `greet` Tool，接收名字並回傳問候語
3. 用 MCP Inspector 測試呼叫這兩個 Tool

**[延伸]**
1. 新增一個 `calculate` Tool，支援加減乘除四種運算
2. 為 Tool 加入輸入驗證，回傳友善錯誤訊息

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Tool 未出現在選單 | Server 未正確回應 tools/list | 確認 `mcp.AddTool()` 正確執行 |
| 輸入參數解析失敗 | struct tag 格式錯誤 | 檢查 `json:"xxx"` 與 `jsonschema:"xxx"` |
| 回傳結果為空 | Handler 未正確建構 CallToolResult | 確認回傳 `&mcp.CallToolResult{Content: [...]}` |

#### ⑥ 驗收標準（DoD）

- [ ] 在 Inspector 中能看到 add、greet Tool
- [ ] 呼叫 add Tool 能正確回傳計算結果
- [ ] 呼叫 greet Tool 能正確回傳問候語

### U04｜Resource Capability（90 min）

#### ① 為什麼先教這個？

Resource 讓 AI 能「讀取資料」。適合提供設定檔、日誌、資料庫內容等讓 AI 分析。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 範例程式碼 | `examples/U04-resources/main.go` | 含靜態與動態 Resource |
| 資料檔案 | `examples/U04-resources/data/config.json` | 範例 JSON 資料 |

#### ③ 核心觀念

1. **Resource 定義**：URI、名稱、描述、MIME type
2. **Static vs Dynamic**：靜態資源（檔案）vs 動態資源（即時計算）
3. **mcp.AddResource()**：註冊 Resource
4. **Resource URI 格式**：`file://`、`custom://` 等
5. **ReadResourceResult**：回傳內容與 MIME type

#### ④ 動手做

**[必做]**
1. 新增一個靜態 Resource，提供專案 README 內容
2. 新增一個動態 Resource，回傳當前系統時間
3. 在 Claude Desktop 測試讀取這些 Resource

**[延伸]**
1. 新增一個 Resource 讀取指定目錄的檔案列表
2. 為 Resource 加入快取機制

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Resource 內容亂碼 | MIME type 設定錯誤 | 確認使用正確的 MIME type（如 `text/plain`、`application/json`） |
| 讀取檔案失敗 | 路徑為相對路徑 | 改用絕對路徑或相對於執行目錄的路徑 |

#### ⑥ 驗收標準（DoD）

- [ ] Claude 能讀取 README Resource 並摘要內容
- [ ] Claude 能讀取時間 Resource 並告知當前時間

### U07｜HTTP Transport（90 min）

#### ① 為什麼先教這個？

Stdio 適合本地開發與 Claude Desktop；HTTP 適合部署到遠端伺服器，讓多個 Client 共用。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 範例程式碼 | `examples/U07-http-transport/main.go` | HTTP 模式 Server |
| 測試腳本 | `examples/U07-http-transport/test.sh` | curl 測試指令 |

#### ③ 核心觀念

1. **Transport 抽象**：MCP 支援不同傳輸層（Stdio、HTTP、WebSocket）
2. **HTTP 模式限制**：Go SDK 目前需自行包裝
3. **JSON-RPC over HTTP**：POST 請求，Content-Type: application/json
4. **CORS 設定**：跨域請求處理
5. **Session 管理**：HTTP 模式下的狀態管理

#### ④ 動手做

**[必做]**
1. 將 U05 的 Server 改為 HTTP 模式
2. 使用 curl 測試 JSON-RPC 請求
3. 驗證 Tool 呼叫能正常運作

**[延伸]**
1. 加入基本認證（API Key）
2. 部署到雲端服務（如 Cloud Run）

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| 連線被拒絕 | Port 被佔用或防火牆阻擋 | 換 Port 或檢查防火牆設定 |
| CORS 錯誤 | 未設定 CORS headers | 加入 `Access-Control-Allow-Origin` 等 headers |

#### ⑥ 驗收標準（DoD）

- [ ] Server 在 `localhost:8080` 啟動
- [ ] `curl` 能成功呼叫 Tool

### U08｜Server 認證（90 min）

#### ① 為什麼先教這個？

HTTP 模式讓 Server 可透過網路存取，但也帶來安全風險。認證機制能驗證請求者身份，保護 API 不被濫用。

#### ② 對應素材

| 類型 | 路徑/連結 | 說明 |
|------|----------|------|
| 範例程式碼 | `examples/U08-auth/main.go` | 認證範例 Server |
| 範例程式碼 | `examples/U08-auth/middleware.go` | 認證 middleware |
| 測試腳本 | `examples/U08-auth/test.sh` | curl 測試指令 |

#### ③ 核心觀念

1. **認證 vs 授權**：認證確認身份，授權確認權限
2. **Basic Auth**：帳密驗證，需搭配 HTTPS
3. **Bearer Token**：Token-based 驗證，業界常用
4. **Middleware Pattern**：包裝 handler 處理橫切關注點
5. **Timing Attack 防護**：使用 constant-time comparison
6. **SDK 支援**：`mcp.RequireBearerToken` middleware

#### ④ 動手做

**[必做]**
1. 實作 Basic Auth middleware
2. 實作 Bearer Token middleware
3. 使用環境變數切換認證模式
4. 用 curl 測試不同認證場景

**[延伸]**
1. 實作 Scope-based 存取控制
2. 處理 CORS + Authentication

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Basic Auth 帳密被攔截 | 未使用 HTTPS | 生產環境必須啟用 HTTPS |
| Timing Attack | 使用 `==` 比較密碼 | 改用 `subtle.ConstantTimeCompare` |
| CORS preflight 401 | OPTIONS 請求被認證擋掉 | CORS middleware 放在 Auth 之前 |

#### ⑥ 驗收標準（DoD）

- [ ] Basic Auth 錯誤帳密回傳 401
- [ ] Bearer Token 無效回傳 401
- [ ] 正確認證後能呼叫 Tool
- [ ] Health Check 不受認證影響
